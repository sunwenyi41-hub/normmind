# Feature Specification: 管理后台与质量运营

**Feature Branch**: `001-admin-quality-ops`  
**Created**: 2026-07-05  
**Status**: In Progress  
**Input**: 区分普通用户与管理员，保护后台路由，并逐步将反馈、QA 样本和评测从演示数据切换为可维护数据。

## User Scenarios & Testing

### User Story 1 - 安全访问管理后台 (Priority: P1)

管理员使用授权邮箱登录后可看到并访问管理后台；普通用户看不到入口，直接访问后台地址也会被拦截。

**Why this priority**: 后台包含全局质量与用户反馈信息，权限边界是其他功能的前置条件。

**Independent Test**: 分别使用管理员和普通用户登录，验证入口可见性与 `/admin` 直接访问结果。

**Acceptance Scenarios**:

1. **Given** 已授权的管理员，**When** 登录工作台，**Then** 可看到并进入管理后台。
2. **Given** 普通用户，**When** 登录工作台，**Then** 不显示管理后台入口。
3. **Given** 普通用户，**When** 直接访问 `/admin`，**Then** 被服务端拦截并返回工作台。

### User Story 2 - 处理真实用户反馈 (Priority: P2)

管理员可查看真实问答反馈，按评价与时间进行筛选，并记录处理状态。

**Independent Test**: 提交一条无用反馈，管理员能在后台看到并更新状态，普通用户无法读取其他人反馈。

### User Story 3 - 维护 QA 与评测记录 (Priority: P3)

管理员可新增、编辑 QA 样本，并查看评测执行结果，为 Agent 调优提供可追踪依据。

**Independent Test**: 新建一条 QA 样本并关联一次评测结果，刷新后数据仍可查看。

### Edge Cases

- 管理员邮箱大小写或前后存在空格时，授权判定仍应正确。
- 用户权限变更后，新会话必须使用最新权限。
- 后台数据为空时，页面应显示可操作的空状态。

## Requirements

### Functional Requirements

- **FR-001**: 系统必须在服务端判定管理员权限。
- **FR-002**: 系统必须向普通用户隐藏管理入口。
- **FR-003**: 未登录或非管理员不得访问任何管理子页。
- **FR-004**: 授权信息必须来自普通用户无法篡改的来源。
- **FR-005**: 管理员必须能查看并处理真实用户反馈。
- **FR-006**: 反馈、QA 样本和评测数据必须具有明确的访问边界。
- **FR-007**: 后台数据操作失败时必须显示可理解的错误与重试方式。

### Key Entities

- **Administrator**: 拥有后台权限的已登录用户。
- **Feedback Case**: 用户对某条回答的评价、原因与处理状态。
- **QA Sample**: 用于验证规范问答质量的问题、期望要点与分类。
- **Evaluation Run**: 一次评测执行及其通过率、失败项与时间。

## Success Criteria

- **SC-001**: 100% 的非管理员后台访问测试被拦截。
- **SC-002**: 管理员在登录后 2 步内进入后台。
- **SC-003**: 新提交的反馈在 5 秒内可于后台查看。
- **SC-004**: 普通用户无法查看或修改他人的反馈、QA 与评测数据。

## Assumptions

- 首版使用服务端邮箱白名单，同时兼容受信任的管理员角色声明。
- 用户继续使用现有邮箱登录，不单独建设管理员登录页。
- 后台操作量以小规模内测为主。
